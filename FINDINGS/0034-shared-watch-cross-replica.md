# FINDINGS: 0034-shared-watch-cross-replica

## What we were trying to learn

Whether multiple replicas of a library-mode aggregated apiserver can
serve consistent Watch streams by all watching the same CRD store on
the host cluster. Specifically: if a write hits replica B, does a
watch client connected to replica A see the event? And does
cross-replica watch resume work when all replicas share a single
etcd-assigned resourceVersion stream?

## What we did

Built a 3-replica StatefulSet library-mode AA
(`widgets.aggexp.io/v1`) backed by a namespace-scoped CRD
(`widgetstorages.aggexpstorage.aggexp.io/v1`) on the host
kube-apiserver. Each replica runs an independent dynamic informer
against the CRD. Informer events fire into a local
`watch.Broadcaster` via a custom `shared.REST` adapter that
bypasses the substrate's per-replica `atomic.Uint64` RV counter
and instead passes through the host CRD's etcd-assigned
resourceVersion unchanged.

Per-pod Services (`aggexp-0`, `aggexp-1`, `aggexp-2`) allow
replica-pinning by patching the APIService's `spec.service.name`.
TLS cert SANs include all per-pod service DNS names.

Tested six scenarios: basic CRUD, cross-replica watch propagation,
list consistency, watch resume against a different replica,
concurrent writes, and cross-replica event latency measurement.

## What we observed

**Cross-replica watch propagation: works.** A watch registered
against replica 0 (via APIService pin) received ADDED events for
writes that went through replica 1 (after re-pinning). The
informer→broadcaster path on the watcher-side replica fires within
~4ms of the write hitting etcd. All three replicas' informers
observe the same event within <1ms of each other (they share the
same etcd watch stream).

**List consistency: perfect.** All three replicas return identical
`listRV` values (the high-water-mark of host CRD RVs observed by
the informer), identical object sets, and identical per-object RVs.
This follows trivially from the fact that Get/List read from the
informer cache, which is populated by the same etcd watch stream.

**Watch resume against a different replica: works without 410.**
RV obtained from replica 0's list was passed as
`?resourceVersion=1024` in a watch opened against replica 2. No 410
Gone was returned. The adapter replays existing list-state as ADDED
prefix events, then streams new events. This is the design choice
in `pkg/shared/rest.go`: tolerate any host RV, never 410, always
replay current state. Cost: a stale resume re-receives objects the
client already has as prefix ADDED events. Benefit: never silently
misses events, and cross-replica resume always succeeds.

**Concurrent writes: all succeed for distinct objects.** 10
concurrent `kubectl create` calls (load-balanced across replicas)
all succeeded with zero conflicts because the backing CRD uses
etcd's optimistic concurrency per-object. Same-object concurrent
writes from different replicas would produce 409 Conflict from
kube-apiserver's CRD machinery — exactly the expected behavior
without any in-AA locking.

**Cross-replica event propagation latency: ~4ms.** Measured from
replica 1's `create` log timestamp to replica 0's `informer-event`
log timestamp: 3.8ms. The path is: AA replica 1 → dynamic client
Create → kube-apiserver CRD handler → etcd write → etcd watch
stream → kube-apiserver notifies informer watch → informer on each
replica dispatches event. All three replicas' informers fire within
<1ms of each other.

## What surprised us

**The latency is dominated by etcd, not by the informer.** The
~4ms total includes the kube-apiserver → etcd round trip + etcd →
watch-stream notification. The informer dispatch on the receiving
replica adds essentially zero additional latency (<1ms delta between
all three replicas receiving the same event). This means the
informer-per-replica architecture doesn't add meaningful fan-out
overhead.

**TLS SANs for per-pod Services are a non-obvious requirement.**
The cert generated by `hack/gen-certs.sh` only had SANs for the
load-balanced service name (`aggexp.aggexp-system.svc`). When
pinning the APIService to `aggexp-0`, kube-apiserver's aggregation
layer connects to `aggexp-0.aggexp-system.svc` and TLS verification
fails. The fix was regenerating certs with all per-pod service DNS
names in the SAN list. This is a consequent of the replica-pinning
test approach, not a fundamental finding.

**The "no 410 on resume" choice has a concrete observable cost.**
When a watch resumes with a known RV, the adapter replays all
current objects as ADDED prefix events regardless of whether those
objects are newer or older than the requested RV. A client that
already has `alpha` and `cross-test-1` will see them re-delivered.
For a `SharedInformer` client (which deduplicates via its local
store), this is invisible. For raw `kubectl get -w` it means a
visual repeat of objects on reconnect. The alternative (410 Gone
forcing re-list) is strictly worse for cross-replica resume because
it forces the client to list+watch from scratch anyway.

## Which fundamentals this touches

**Watch and consistency semantics (primary).** This is the first
experiment to demonstrate consistent cross-replica watch under the
library-mode substrate. The host CRD's etcd-assigned RV is the
unifying abstraction: because all replicas observe the same
monotonic RV stream, they can serve list and watch responses that
agree. Per-replica `atomic.Uint64` RVs (the substrate default)
would make this impossible — each replica's counter progresses
independently, so cross-replica resume is semantically undefined.

**Storage independence (secondary).** The "CRD IS the storage"
pattern (variant of the 0010 CRD-facade axis) composes cleanly
with multi-replica deployment. The CRD provides: (a) persistence
the AA doesn't need to implement, (b) optimistic concurrency via
`metadata.resourceVersion` on Update, (c) a single RV authority
from etcd that all informers observe consistently, (d) host-cluster
RBAC on the backing resource as an additional security layer. The
cost vs. in-memory storage: one extra kube-apiserver hop per write
(~4ms at lab scale), and the informer watch overhead per replica
(negligible at <100 objects).

## Consequents

- TLS cert SANs must include per-pod service DNS names when using
  APIService pinning for replica-level routing. The aggregation layer
  always dials the service name from the APIService spec.
- Informer resync period (30s in this experiment) fires MODIFIED
  events with the same RV for all cached objects. These are
  observable in logs and are dispatched through the broadcaster.
  A production deployment might set resync to 0 and rely purely on
  the watch stream, or filter duplicate-RV MODIFIEDs in the event
  handler.
- The `kubectl get -o json` list-level `metadata.resourceVersion` is
  returned correctly by the raw API but appears empty in some kubectl
  output formatters (jsonpath template `{.metadata.resourceVersion}`
  on a list returns empty). This is a kubectl consequent, not an AA
  behavior issue.

## What this changes for SYNTHESIS and EXPERIMENTS

For SYNTHESIS: the Watch and consistency semantics section should note
that host-CRD RV authority enables consistent cross-replica watch
and cross-replica resume without any inter-replica coordination. The
per-replica `atomic.Uint64` (current substrate default) is
fundamentally incompatible with multi-replica consistent watch. A
future substrate change should offer a "pass-through RV" mode for
backends that supply their own monotonic RV (CRD-backed, etcd-backed,
sequence-backed).

For EXPERIMENTS: 0034's finding that CRD-RV-authority eliminates
the need for 0032/0033's locking for *read consistency* is
significant. Locking from 0032/0033 is still needed for
*write-side ownership coordination* (e.g. "only one replica should
process a reconcile for this object"), but for the "all replicas
serve the same data consistently" problem, shared-CRD + informer is
sufficient without any additional coordination.

## Open questions

- Does the informer resync cause visible jitter for real
  SharedInformer clients (controllers that key on RV change)?
  The resync fires MODIFIED with unchanged RV; client-go's
  reflector should treat this as a no-op (same RV = no new event
  in the DeltaFIFO). Worth validating under 0012-style sustained
  controller-runtime load.
- What happens at 10k+ objects? The "replay list as prefix" design
  on watch resume would replay 10k ADDED events on every
  reconnect. At scale, a ring-buffer of recent events (keyed on
  host-CRD RV) that enables true incremental resume would be
  necessary.
- Can the substrate's `runtime/storage.REST` be extended with a
  "pass-through RV" mode that backends opt into, rather than
  requiring a parallel `shared.REST` implementation? This would
  make the 0034 pattern reusable without code duplication.
- SSA behavior under this pattern: untested in 0034 (scope cut).
  The backing CRD carries managedFields natively, so SSA should
  work through the 0010-style apiVersion rewrite, but multi-replica
  concurrent SSA with field managers is uncharacterized.
