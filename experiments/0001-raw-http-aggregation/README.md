# Experiment 0001: raw-http-aggregation

A hand-rolled Go `net/http` server that pretends to be a Kubernetes
aggregated API server for `hellos.aggexp.io/v1`. Uses only Go stdlib. No
`k8s.io/apiserver`, no dependencies. It exists to probe, not to be
production-ready.

## Hypothesis

What is the minimum wire contract the kube aggregation layer + kubectl
actually demand of an aggregated API server? Can we serve a "working"
aggregated API — `kubectl api-resources`, `kubectl get`, `kubectl get -w`,
`kubectl explain` — without pulling in `k8s.io/apiserver` at all?

Specifically we want to see:

- Which discovery endpoints (`/apis`, `/apis/<group>`, `/apis/<group>/<ver>`)
  the aggregator hits on us and which ones kubectl hits through it.
- Which OpenAPI endpoints kubectl demands for `explain` to succeed, and
  how strict it is about their content (v2 vs v3 negotiation).
- What content-type / Accept dance kubectl does for `get` (JSON vs Table).
- How watch behaves when we cut corners (no real resourceVersion, only
  BOOKMARK events, no true change stream).
- What X-Remote-* headers the aggregator forwards — identity handoff.

## What this is

A single `main.go` that serves HTTPS on :8443 with a serving cert mounted
from the `aggexp-serving-cert` Secret. It:

- Returns static discovery for group `aggexp.io` / version `v1` with one
  resource `hellos` (cluster-scoped, singular `hello`, shortname `hi`).
- Returns a fixed 2-item `HelloList` (static RVs 1 and 2).
- Handles `Accept: application/json;as=Table;v=v1;g=meta.k8s.io` by
  returning a `meta.k8s.io/v1 Table`.
- Handles `?watch=true` with chunked NDJSON: initial `ADDED` events for
  each static item, then a `BOOKMARK` every 10s until the client
  disconnects. Deliberately partial — no real change stream.
- Serves minimal OpenAPI v2 and v3 documents.
- Does **not** verify the kube-apiserver's aggregator client cert. Probe
  goal is observation, not authz. All incoming headers (X-Remote-*,
  User-Agent, Accept) are logged per request.

## How to run

From repo root, assuming `kind`, `kubectl`, `go`, `docker` are installed:

```
./hack/gen-certs.sh
./hack/make-kind.sh
./hack/deploy.sh deploy/manifests   # base: namespace, SA, RBAC, Service, APIService, cert Secret

# Build and load the probe image into the kind cluster.
docker build -t aggexp-probe-0001:dev experiments/0001-raw-http-aggregation/
kind load docker-image aggexp-probe-0001:dev --name aggexp

# Apply the override Deployment (swaps the base Deployment's image).
./hack/deploy.sh experiments/0001-raw-http-aggregation/manifests

kubectl -n aggexp-system rollout status deploy/aggexp
kubectl api-resources | grep hellos
kubectl get hellos
kubectl get hellos -w   # then Ctrl-C after a bookmark or two
kubectl explain hello
./hack/test-compat.sh
```

Deploy order matters: the base `deploy/manifests/` dir provides the
Namespace, ServiceAccount, RBAC, Service, APIService, and serving-cert
Secret that this overlay assumes already exist. This experiment's
`manifests/` dir contains ONLY the Deployment override.

## Decisions made

- **Bookmark interval: 10s.** Short enough to see a few during a manual
  `kubectl get -w`, long enough to not spam logs.
- **Static resourceVersions 1 and 2 for items; list RV 1.** Intentionally
  stale / non-monotonic — part of the probe is seeing whether clients
  care.
- **BOOKMARK RV seed 10, atomically incremented per bookmark.** Separate
  counter from item RVs, so we can tell bookmark RVs apart from item RVs
  in logs.
- **No mTLS verification.** `tls.NoClientCert`. The probe needs to see
  what the aggregator sends, not reject it.
- **Single-file stdlib-only Go.** No `k8s.io/*` deps. The whole point is
  to find out what we really need before we pull in a framework.
- **Own prefix handler at `/apis/aggexp.io/v1/` for item paths** instead
  of relying on Go 1.22 ServeMux path patterns — simpler and version-
  agnostic.
- **Chunked watch uses `json.Encoder` which appends `\n` per object**,
  giving NDJSON framing without manual delimiter writes.

## Prerequisites

- kind cluster `aggexp` created via `hack/make-kind.sh`.
- Serving cert generated by `hack/gen-certs.sh` and loaded into the
  `aggexp-serving-cert` Secret by the base deploy.

## What we're looking to learn

- **Wire protocol fidelity.** Which endpoints + content types must we get
  right for kubectl / aggregator to accept us? Where's the minimum bar?
  What breaks first when we cheat?
- **Watch and consistency semantics.** What does kubectl do when the
  server emits only ADDED + BOOKMARK and never a MODIFIED/DELETED?
  Does it re-list on a stale RV? Does it reconnect? How does it react to
  a BOOKMARK whose RV is disjoint from item RVs?
