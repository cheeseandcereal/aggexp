# Experiment 0007: runtime-fs-driver

A filesystem-backed aggregated apiserver that exposes the contents of
a server-side directory as `files.aggexp.io/v1` resources. Read-only.
Built atop the substrate introduced by this branch (`runtime/`), not
from scratch.

This experiment serves two purposes:

1. **Demonstrate the substrate.** If the extraction worked, writing
   0007 should be substantially shorter than 0002 or 0004 — and the
   things that were shared boilerplate in those experiments should
   now live in `runtime/`.
2. **Probe resource-modeling freedom.** A filesystem is a concrete,
   well-understood substrate; asking "are files a reasonable thing
   to project as a Kubernetes resource?" exercises the same axis
   that 0004 did with GitHub repos but with a different shape.

## Hypothesis

- Writing a new aggexp-style AA against the substrate reduces the
  per-experiment boilerplate meaningfully (target: ≥30% line-count
  reduction vs. 0004's equivalent code paths).
- A `Backend` interface suffices to capture the shape shared by the
  Hello / Repo / File types without overspecification.
- Hidden-file filtering, filename sanitization, and diff-based
  polling carry cleanly from 0004's pattern; no substrate changes
  needed.

## What the substrate gave us

- `runtime/server` — generic Options, flags, Config, Run. The
  experiment-side `pkg/server/server.go` is ~100 lines instead of
  ~225 in 0004.
- `runtime/storage` — full rest.Storage adapter over a Backend. The
  experiment-side `pkg/fsbackend/backend.go` is just the backend
  logic (~230 lines including sanitization + poll loop); it does
  not re-implement rest.Storage / watch fan-out / RV accounting.
- `runtime/authz` — external policy authorizer. Not exercised in
  this experiment (no policy-service deployment).
- `runtime/group` — API-group install helper.

The experiment still owns (per AGENTS.md's rule that scheme-bound
code belongs with the experiment):

- `pkg/apis/aggexp` + `pkg/apis/aggexp/v1` — type definitions,
  register, conversion.
- `pkg/apis/aggexp/install` — scheme wiring.
- `pkg/apiserver` — Scheme + Codecs + unversioned types init.

## How to run

From the repo root:

```
./hack/gen-certs.sh
kind create cluster --name aggexp-runtime
kubectl config use-context kind-aggexp-runtime
kubectl create namespace aggexp-system

./hack/deploy.sh deploy/manifests

docker build -t aggexp-files:dev \
  -f experiments/0007-runtime-fs-driver/Dockerfile .
kind load docker-image aggexp-files:dev --name aggexp-runtime

AGGEXP_IMAGE=aggexp-files:dev \
  ./hack/deploy.sh experiments/0007-runtime-fs-driver/manifests

kubectl -n aggexp-system rollout status deploy/aggexp

# Wait ~10s for first scan, then:
kubectl get files
kubectl get files -o yaml
```

## Decisions made

- **Scope cut: no fsnotify.** The experiment polls on a 5s ticker
  like 0004 rather than wiring inotify. Adding fsnotify is a
  one-liner (`go get github.com/fsnotify/fsnotify`) but not worth
  the extra scope for a substrate-validation experiment.
- **Scope cut: read-only.** Writing files back via `kubectl apply`
  is left to a later experiment (name collisions with path
  semantics, permissions, directory creation, etc. all deserve
  their own probe).
- **OpenAPI: hand-edited from 0004's generated file.** We did not
  re-run openapi-gen (it is heavy code-gen tooling and the
  substrate extraction was already the meat of this experiment).
  Instead the 0004 generated file was copied, the module paths and
  4 schema-function bodies were rewritten for File/FileList/
  FileSpec/FileStatus, and the rest (meta/v1 + runtime + version
  schemas) was kept verbatim. Treat `zz_generated.openapi.go` as
  "hand-rolled from a generated baseline" for this experiment.
- **Only top-level files are listed.** Subdirectories are skipped.
  Nested trees would force a naming decision (slashes are not
  legal in Kubernetes names) better handled by a dedicated probe.
- **Hidden files filtered.** Names starting with `.` are skipped.
  Consistent with `ls` defaults and prevents noise from editor
  swap files.
- **Filename sanitization: lowercase, alphanumerics + `-` + `.`
  only.** Everything else becomes `-`. Collisions are possible but
  rare; a later experiment could probe.
- **Sample data lives in a ConfigMap.** The Deployment mounts it
  at `/etc/aggexp-sample-files`. This keeps the experiment
  self-contained (no host-path mounts, no init container).

## Prerequisites

- A kind cluster named `aggexp-runtime` (not the shared `aggexp`
  one — kept distinct so the substrate test can coexist with
  0002/0003/0004 runs on the default cluster).
- Serving cert generated by `hack/gen-certs.sh`.
- Base manifests applied via `hack/deploy.sh deploy/manifests`.

## What we're looking to learn

- **Resource modeling freedom.** Does the `Backend` abstraction
  tolerate a new shape (files) without the substrate needing
  changes?
- **Substrate line-count reduction.** Is the experiment-side code
  meaningfully smaller than 0002 / 0004 for the same capability?
- **Interface ergonomics.** Are any abstraction seams awkward when
  actually consumed? Every awkward seam is a FINDING.
